当前项目属于“已有较完整工程实现，但答辩前仍需明显补强”的状态。代码层面并非简单页面壳子，能够看到 Vue 前端、Spring Boot 后端、MyBatis 数据访问、模型配置、用例生成、自动评估、再次生成、批量生成、生成记录、用例分析和 coverage.py 覆盖率统计等核心模块；论文和 PPT 主线也基本围绕这些模块展开。但是，本地材料中缺少 README、部署说明、完整数据库初始化脚本、测试代码、实验原始数据导出文件；云端前端能访问，但后端接口在检查时无法完成登录验证；论文存在封面/承诺书信息未填、文献与引用字段需核查、测试证据不够实、部分章节 AI 味较强等问题。因此不建议按“已经完全达标”处理，应按答辩前重点整改项目继续补齐证据链。

## 5. 代码与系统实现检查结果

### 已有基础

- 后端具备基本分层：`controller`、`service`、`mapper`、`entity`、`resources/mapper`、`resources/sql`。
- 主要接口模块较全：`GenerateController`、`FunctionInfoController`、`RequirementSpecController`、`ModelConfigController`、`GenerateRecordController`、`AdminController`、`UserController`。
- 核心业务集中在 `GenerateService`：可见生成提示词、模型调用、结果规范化、记录保存、自动评估、批量生成、再次生成、公开测试基准对比等逻辑。
- `CoverageService` 实现了 Python 测试代码提取、临时文件执行、coverage.py JSON 解析、`SUCCESS`/`TEST_FAILED`/`FAILED`/`SKIPPED` 状态处理。
- 前端页面覆盖核心功能：登录、首页、用例生成、生成记录、用例分析、批量生成、函数管理、需求规格说明书管理、模型配置、管理员/用户管理。
- 用户手册说明了系统操作流程，能作为基础归档材料之一。

### 存在问题

- 本地缺少 `node_modules`，未执行前端构建；后端虽然有 Maven 环境，但没有本地数据库和真实配置，未做本地联调验证。
- `application.yml` 使用占位数据库账号、密码和 API Key；这能避免泄露密钥，但缺少可复现部署说明。
- 数据库初始化不完整。`GenerateRecordSchemaInitializer` 和 SQL 文件只确保 `function_info`、`generate_record`，但 `admin`、`user`、`model_config`、`requirement_spec` 没有完整建表脚本。
- 明文密码校验，无密码哈希；毕业设计可作为一般原型通过，但论文和答辩中不要夸大安全性。
- 无后端单元测试、接口测试、前端测试、覆盖率报告或启动日志。
- 根目录没有 README，答辩前检查人员无法快速知道如何启动前端、后端、数据库和云端演示。
- 批量实验和模型调用依赖外部模型服务，如果模型配置为空或后端不可用，演示会受阻。

## 10. 论文 AI味检查结果

### AI味较强的章节或部分

- 摘要和英文摘要：信息覆盖全面，但单段承载过多内容，表达非常整齐，像把背景、问题、方法、功能、结果一次性压缩生成。
- 1.1 研究背景和意义、1.2 国内外研究现状：高频出现“已有研究表明”“相关研究指出”“能够为后续提供基础”等平滑连接词，文献叙述多，和本系统的具体代码、数据、失败边界连接不够。
- 2.1 相关技术基础、2.3 可行性分析：技术栈介绍较标准化，读起来像通用 Web 管理系统模板加选题关键词。
- 5.2 功能测试、5.4 用例分析结果讨论：测试表全部通过，结论完整，但缺输入、输出、日志、记录编号和原始导出文件支撑，容易形成“测试写得完整但证据偏弱”的观感。
- 6.1 全文总结、6.2 后续工作展望：总结非常完整，覆盖多语言、复杂代码、模型优化、需求解析等方向，但与当前系统实际限制的证据连接还不够。

### 具体问题表现

- 模板化：章节小结、可行性分析、总结展望采用常见论文句式，缺少本项目独有细节。
- 空泛化：多处使用“提高质量”“提供数据基础”“降低成本”“便于后续分析”，但未给出对应记录、指标或附件。
- 拼接感：研究现状中多个文献点连续出现，但每篇文献如何影响本系统设计说得不够具体。
- 概念堆砌：大语言模型、链式思维、覆盖率反馈、提示工程、模型评估等概念较多，部分段落没有落到代码路径或运行结果。
- 缺少实现细节：第 4 章虽提到 `GenerateService`、`CoverageService`，但可再补接口路径、核心方法、关键字段和异常处理流程。
- 缺少真实测试支撑：第 5 章缺少生成记录 ID、样例函数、生成结果片段、coverage.py 输出、失败样例和原始数据附件。
- 总结与证据不匹配：结论说系统已完成多项能力，但部署后端当前不可验证，本地也缺可复现运行说明。

### 判断依据

- 代码中确实存在真实模块，所以不能简单判断论文与系统脱节。
- AI 味主要来自文字表达过于平滑、结论过于完整、测试证据不足，而不是因为没有系统实现。
- 论文越靠近代码实现的部分，AI 味越弱；越靠近背景、研究意义、可行性、总结展望，AI 味越强。

### 风险说明

- 答辩老师如果追问“表 5.4 数据从哪里来”“coverage.py 具体跑了哪个函数”“失败记录如何处理”“为什么说链式分析更好”，当前本地材料不足以快速给出证据。
- 若最终 PDF 中出现 Word 域代码或引用错误，会被认为格式检查不认真。
- 若云端后端不可用，论文中“可以运行、可以演示”的说法会被现场验证削弱。

### 修改建议

- 每个核心结论后补一个证据：记录 ID、截图编号、接口名、数据库字段、覆盖率数值、导出文件名。
- 第 5 章增加一个完整案例表：函数名、源码摘要、模型、策略、run_index、生成用例数量、评估分、coverage_status、line_coverage、branch_coverage。
- 增加失败或跳过案例：如非 Python 函数触发 `SKIPPED`，不可执行测试触发 `FAILED`，断言失败触发 `TEST_FAILED`。
- 把“显著提升”“有效提升”等强结论改为“在本次记录中平均评估分更高”“在样例中覆盖率达到某值”。
- 减少泛泛文献综述，每个小节保留 1 到 2 句说明其对系统设计的直接影响。
- 摘要和总结中明确当前边界：主要面向函数级代码，coverage.py 只覆盖 Python unittest，其他语言仍需扩展。

### AI味相对较弱、较贴近真实工作的部分
